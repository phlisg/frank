# Frank — Claude Notes

## Task Tracking

This project uses `td` for task tracking. A `.todos` folder is present — always use `td` commands (`td create`, `td close`, `td review`, etc.) for any task tracking work, in addition to the native task tools used by superpowers subagent-driven development.

## What Frank Is

Frank is a Go CLI that scaffolds and manages Laravel Docker environments. It reads `frank.yaml` in the user's Laravel project and generates everything into `.frank/` (Dockerfile, compose.yaml, Caddyfile/nginx config). The `docker compose` project lives in `.frank/`, not the repo root.

## Architecture

```
cmd/              cobra commands (activate, init, generate, up, down, add, remove, worker, ps, …)
internal/
  compose/        generates .frank/compose.yaml by merging template fragments
  config/         frank.yaml schema, validation, defaults
  docker/         thin docker compose CLI wrapper (always uses -f .frank/compose.yaml)
  shell/          alias table for frank activate/deactivate
  template/       template engine (renders from templates/ FS)
  watch/          host-side fsnotify watcher (walker, classifier, debouncer, dispatcher, first-run suppression, pidfile lifecycle + already-running guard, orphan detection, Daemonize helper done; CLI + up/down integration still open — see td)
templates/
  runtimes/       frankenphp/, fpm/ — each has compose.fragment.tmpl + Dockerfile.tmpl
  services/       one dir per service — compose.fragment.tmpl + env.tmpl
  workers/        schedule.fragment.tmpl, queue.fragment.tmpl, init.fragment.tmpl (laravel.migrate)
```

## Critical Invariant

**Every `docker compose` invocation must include `--project-directory . -f .frank/compose.yaml`.**  
This applies in both `internal/docker/docker.go` (programmatic calls) and `internal/shell/shell.go` (user-facing aliases). The `dc` constant in shell.go owns this for aliases.

## Compose Generation

`Generator.Generate()` in `internal/compose/compose.go`:
1. Render runtime fragment (`templates/runtimes/<runtime>/compose.fragment.tmpl`)
2. Render + merge each service fragment (`templates/services/<service>/compose.fragment.tmpl`)
3. Inject `depends_on` into `laravel.test` — done post-merge in Go, because templates don't know which services are selected
4. Validate port uniqueness
5. Marshal to YAML with a "Generated by Frank" header

Rule: any logic that needs to know the full service list (e.g. `depends_on`, port validation) belongs in `Generate()`, not in templates.

## Services

Valid: `pgsql`, `mysql`, `mariadb`, `sqlite`, `redis`, `memcached`, `meilisearch`, `mailpit`  
Only one database allowed. Defaults: `pgsql` + `mailpit`.

Healthchecks present (use `service_healthy`): pgsql, mysql, mariadb, redis, meilisearch, mailpit  
No healthcheck (use `service_started`): memcached

The `serviceDepends(cfg)` helper in `internal/compose/compose.go` owns the `depends_on` map construction. Reused for both `laravel.test` and the workers' `laravel.migrate` init service — if you add another service type, update that helper rather than duplicating.

## Workers

See `docs/superpowers/specs/2026-04-18-workers-schedule-queue-design.md` for the full design.

Config: `workers.schedule` (bool) + `workers.queue[]` (list of pools). Defaulting in `internal/config/config.go` applies `queues: ["default"]` first, then derives `name` from `queues[0]`. Unknown keys warn but do not error. Pool names match `[a-z0-9_-]+` and must be unique.

Generator in `internal/compose/compose.go` `emitWorkers`:
1. Render `templates/workers/init.fragment.tmpl` once → `laravel.migrate` (one-shot: composer install when vendor/ missing, then artisan migrate, then exits).
2. If `workers.schedule`: render `schedule.fragment.tmpl` → `laravel.schedule`.
3. For each pool × count: render `queue.fragment.tmpl` → `laravel.queue.<pool>.<i>`.
4. Post-merge injections (all of which stay in Go, not templates, so fragments remain runtime-agnostic):
   - Copy `laravel.test`'s `build:` block onto `laravel.migrate` and every worker so compose builds the image once via tag-dedup — without this, compose tries to pull `frank-<project>-laravel.test` from docker.io.
   - Inject db `depends_on` onto `laravel.migrate` (via `serviceDepends`).
   - When `cfg.PHP.Runtime == "fpm"`: set `user: sail` on `laravel.migrate` and every declared worker. php-fpm's in-pool user drop does not apply to CLI commands, so without this, workers' `queue:work` runs as root and corrupts bind-mount ownership.

Worker fragment invariants (in the templates themselves):
- `container_name: <service-name>` — pins clean names (`laravel.schedule`, `laravel.queue.<pool>.<i>`) rather than compose's autogenerated `<project>-<svc>-1` form.
- `environment: WWWUSER=${UID:-1000}` — required so the entrypoint's `usermod -u $WWWUSER sail` aligns the sail UID with the host user. Without this, workers race `laravel.test`'s chown on `/var/www/html/storage` and trigger Permission denied on `storage/logs/laravel.log`.
- `healthcheck: disable: true` — workers reuse the laravel.test image but never start Caddy/nginx, so the inherited healthcheck always fails otherwise.
- `depends_on: { laravel.test: service_started, laravel.migrate: service_completed_successfully }` — the migrate dependency is load-bearing: queue:work checks the queue:restart signal via the cache store on startup, and Laravel 11+'s default database cache driver crash-loops without the `cache` table.

Ad-hoc workers (from `frank worker queue|schedule`) are launched via `docker compose run -d --name <name> --label frank.worker=adhoc laravel.test …`. They are **not** in compose.yaml, so `frank down` must clean them explicitly (`docker.Client.AdhocWorkerNames` → `StopContainers`). That cleanup lives in `cmd/down.go`; do not move it back into compose.

## Sail Interop Notes

`vendor/bin/sail` is a **bash script** (not PHP) — `./vendor/bin/sail <cmd>` works without a local PHP install. Keep this in mind when writing user-facing messages or docs that reference Sail commands.

`frank eject` installs Sail using the services configured in `frank.yaml` (reads `cfg.Services`, passes as `--with` to `sail:install`).

## FPM Runtime Notes

The FPM Dockerfile uses `CMD ["php-fpmX.Y", "-F"]` to run in foreground (no-daemonize).  
**Do not add `daemonize = no` to php-fpm.conf** — PHP 8.5 removed this as a valid global directive and FPM will refuse to start with `unknown entry 'daemonize'`.

## Testing

```
go test ./cmd/ -update
```
